# Архитектурная оценка: alertsbot

Ревью с позиции senior architect. Дата: 2026-03-20.

Проект: ~95 строк Python, 3 модуля, 1 endpoint + healthcheck. Оценки калибруются относительно масштаба — микросервис не нуждается в enterprise-паттернах, поэтому «простота = правильная архитектура» в данном контексте.

---

## 1. Separation of Concerns — 8/10

**Что сделано хорошо:**
Три модуля — три ответственности, чёткие и непересекающиеся:

| Модуль | Ответственность |
|---|---|
| `config.py` | Загрузка и валидация конфигурации |
| `telegram.py` | Транспорт — отправка в Telegram API |
| `app.py` | HTTP-слой: роутинг, авторизация, форматирование, оркестрация |

`telegram.py` не знает ни о FastAPI, ни о конфигурации — чистая функция, принимающая примитивы. Это правильная граница.

**Где размыто:**
`app.py` совмещает три роли: HTTP-маршрутизация, бизнес-логика (авторизация + форматирование текста) и оркестрация вызова Telegram. Для 63 строк это нормально, но при росте (новые каналы доставки, шаблоны сообщений) форматирование и авторизацию стоит вынести.

```
app.py:43-44  — авторизация (security concern)
app.py:46-48  — форматирование текста (business logic)
app.py:50-57  — вызов внешнего сервиса (integration)
```

---

## 2. Coupling — 9/10

**Граф зависимостей:**
```
app.py ──→ config.py     (импорт get_settings)
app.py ──→ telegram.py   (импорт send_message)
config.py ──→ (ничего внутреннего)
telegram.py ──→ (ничего внутреннего)
```

Связанность минимальна. Два leaf-модуля (`config`, `telegram`) полностью независимы друг от друга. `app.py` — единственный координатор. Циклических зависимостей нет.

`telegram.send_message()` принимает 5 примитивных аргументов, а не объект `Settings` — это осознанное решение, исключающее связь транспорта с конфигурацией.

**Единственный нюанс:** `settings = get_settings()` вызывается на уровне модуля `app.py:24`. Это связывает момент импорта модуля с наличием `.env`/переменных окружения. Не критично, но затрудняет тестирование без monkey-patching.

---

## 3. Cohesion — 8/10

Каждый модуль фокусирован:

| Модуль | Cohesion | Обоснование |
|---|---|---|
| `config.py` | Высокая | Одна задача: описать схему настроек и предоставить кэшированный accessor |
| `telegram.py` | Высокая | Одна функция, одна задача: отправить текст через Bot API |
| `app.py` | Средняя-высокая | Модель `NotifyRequest` + два route-handler'а + инициализация. Три ответственности в одном файле, но для микросервиса это адекватный размер |

**Снижение:** `app.py` содержит pydantic-модель `NotifyRequest` — это data-layer concern, живущий в HTTP-слое. При добавлении второго endpoint'а модели стоит вынести в `models.py`.

---

## 4. Extensibility — 6/10

**Что легко добавить:**
- Новую env-переменную → одно поле в `Settings` + `.env.example`.
- Новый endpoint → новая функция в `app.py`.

**Что потребует рефакторинга:**

| Сценарий | Проблема |
|---|---|
| Второй канал доставки (Slack, email) | `app.py:50-57` жёстко вызывает `send_message` Telegram. Нет абстракции «канал доставки». Придётся добавлять if/else или вводить интерфейс. |
| Шаблоны сообщений | Форматирование (`app.py:46-48`) вшито в handler. Для разных форматов (HTML, Markdown, per-service шаблон) нужно выносить в отдельный модуль. |
| Retry / queue | Сейчас fire-and-fail. Добавление ретраев потребует либо middleware, либо переработки `send_message`. |
| Несколько чатов | `ALERTS_CHAT_ID` — один. Маршрутизация по сервисам потребует конфиг-маппинга. |

Для текущего scope (один канал, один чат, один формат) — достаточно. Но архитектура не заложена под рост, и это осознанный выбор.

---

## 5. Testability — 5/10

**Сильные стороны:**
- `telegram.send_message()` — чистая async-функция с примитивными аргументами. Легко тестировать с mock httpx через `respx` или `httpx.MockTransport`.
- `Settings` — можно инстанцировать с явными значениями: `Settings(alerts_bot_token="test", ...)`.

**Проблемы:**

| Проблема | Файл:строка | Влияние |
|---|---|---|
| `settings = get_settings()` на уровне модуля | `app.py:24` | При `import alertsbot.app` сразу читается `.env`. Для тестов нужен либо `.env` в test-директории, либо monkey-patch `get_settings`. FastAPI `Depends()` решил бы это. |
| `logging.basicConfig()` на уровне модуля | `app.py:25` | Side-effect при импорте — конфигурирует глобальный root logger. Может конфликтовать с test runner. |
| `@lru_cache` на `get_settings()` | `config.py:29` | Кэш живёт на уровне процесса. Между тестами нужно делать `get_settings.cache_clear()`, иначе все тесты работают с одним инстансом Settings. |
| Нет тестов вообще | — | Ноль файлов `test_*.py`. Нет `pytest` в зависимостях. CI отсутствует. |

**Рекомендация:** Использовать `Depends(get_settings)` в handler'ах для dependency injection. Это стандартный FastAPI-паттерн, который позволяет переопределять settings через `app.dependency_overrides` в тестах.

## 6. Error Boundaries — 7/10

**Текущая модель ошибок:**

```
Клиент → [FastAPI validation] → [Auth check] → [Telegram call] → Ответ
              422                    401              502          200
```

| Граница | Реализация | Оценка |
|---|---|---|
| Невалидный JSON / отсутствие полей | FastAPI/pydantic автоматически → 422 | Хорошо |
| Неверный токен | Явная проверка → 401 | Хорошо |
| Ошибка Telegram | `except Exception` → лог + 502 | Достаточно |
| Пустые секреты при старте | Нет проверки | Плохо |
| Неожиданная ошибка вне try/except | FastAPI default → 500 Internal Server Error | Ок |

**Сильно:** `logger.exception()` в `app.py:59` сохраняет полный traceback — ошибка не теряется.

**Слабо:** Все исключения Telegram (таймаут, 403, 404, DNS-ошибка) маппятся в одинаковый `502 Telegram error`. Клиент не может отличить временную проблему от постоянной (бот заблокирован). Для алертинг-сервиса это особенно неприятно — алерт о поломке самого алертинга не дойдёт, и никто не узнает.

---

## 7. Configuration — 8/10

**Что сделано правильно:**
- Единая точка входа: `Settings` через pydantic-settings.
- Поддержка `.env`-файла и переменных окружения одновременно.
- `@lru_cache` — settings загружаются ровно один раз.
- `extra="ignore"` — лишние переменные в `.env` не ломают старт.
- Все настройки типизированы (`str`, `int`, `float`).

**Что можно улучшить:**

| Проблема | Детали |
|---|---|
| Секреты не обязательные | `default=""` для `ALERTS_BOT_TOKEN`, `ALERTS_CHAT_ID`, `ALERTS_TOKEN`. Сервис стартует без них. |
| Нет валидации значений | `log_level` принимает любую строку (включая `"potato"`). `app_port` принимает 0 или 99999. |
| Путь к `.env` относительный | `env_file=".env"` зависит от CWD процесса. В systemd-юните это решено через `WorkingDirectory=/alertsbot`, но при ручном запуске из другой директории — тихо проигнорируется. |

## Топ-5 архитектурных рекомендаций

### 1. Сделать секреты обязательными (fail-fast)
**Приоритет:** Высокий
**Почему:** Сервис тихо стартует с `ALERTS_BOT_TOKEN=""` — healthcheck зелёный, но ни один алерт не дойдёт. Для alerting-сервиса silent failure — худший сценарий.
**Как:** Убрать `default=""` у `alerts_bot_token`, `alerts_chat_id`, `alerts_token` в `config.py:16-18`. Без `.env` сервис откажется стартовать с понятной ошибкой pydantic.

### 2. Внедрить Dependency Injection для Settings
**Приоритет:** Высокий
**Почему:** Module-level `settings = get_settings()` делает тестирование и динамическую конфигурацию невозможными без monkey-patching. Это главный blocker для написания тестов.
**Как:** Заменить глобальный `settings` на `Depends(get_settings)` в handler'ах. FastAPI нативно поддерживает это, а `app.dependency_overrides[get_settings]` даёт тестам полный контроль.

### 3. Переиспользовать httpx.AsyncClient
**Приоритет:** Средний
**Почему:** Новый TCP+TLS handshake на каждый `/notify` — overhead x10 по сравнению с persistent connection. При всплесках алертов (авария → десятки уведомлений) это станет bottleneck.
**Как:** Создать `AsyncClient` в FastAPI lifespan, передавать в `send_message()` как аргумент. Закрывать в shutdown. Бонус: connection pooling к api.telegram.org.

### 4. Добавить минимальный набор тестов
**Приоритет:** Средний
**Почему:** Ноль тестов → изменения деплоятся вслепую. Для сервиса из 3 файлов достаточно 5-7 тестов, чтобы покрыть все ветки.
**Как:** Добавить `pytest` + `httpx` (для `TestClient`). Минимум: auth 401, успешная отправка (mock Telegram), ошибка Telegram → 502, healthz 200.

### 5. Вынести форматирование сообщения из handler'а
**Приоритет:** Низкий (сейчас), станет Средний при добавлении форматов
**Почему:** Логика форматирования (`app.py:46-48`) вшита в HTTP-handler. Если появятся HTML/Markdown-режимы, per-service шаблоны или локализация — handler раздуется.
**Как:** Выделить функцию `format_alert(payload: NotifyRequest) -> str` в отдельный модуль или в ту же модель. Это также упростит тестирование форматирования без поднятия HTTP-стека.
