# Simplifier-отчёт: alertsbot

Проект маленький и чистый — критических проблем нет. Ниже перечислены точечные находки.

---

## 1. Мёртвый код и неиспользуемые импорты

### 1.1 `from __future__ import annotations` — избыточен на Python 3.11+

| Файл | Строка |
|---|---|
| `alertsbot/app.py` | 3 |
| `alertsbot/config.py` | 3 |
| `alertsbot/telegram.py` | 3 |

**Суть:** Этот импорт нужен для поддержки `str | None` синтаксиса в Python 3.9–3.10. Проект декларирует минимум Python 3.11+, где union-типы работают нативно.

**Рекомендация:** Удалить `from __future__ import annotations` из всех трёх файлов. Поведение не изменится.

### 1.2 `app_host` и `app_port` в Settings — нигде не читаются Python-кодом

| Файл | Строки |
|---|---|
| `alertsbot/config.py` | 20–21 |

**Суть:** Поля `app_host` и `app_port` загружаются в `Settings`, но ни один Python-модуль не обращается к `settings.app_host` / `settings.app_port`. Systemd-юнит читает `${ALERTS_APP_HOST}` и `${ALERTS_APP_PORT}` напрямую из `.env` через `EnvironmentFile=`, минуя pydantic.

**Рекомендация:** Два варианта:
- **Удалить** эти поля, если запуск только через systemd.
- **Оставить**, если планируется `__main__.py`-точка входа типа `uvicorn.run(app, host=settings.app_host, port=settings.app_port)`. Но сейчас такого кода нет.

## 2. Дублирование

### 2.1 `pip install -r requirements.txt` дублируется в restart.sh

| Файл | Строки |
|---|---|
| `scripts/restart.sh` | 7, 9 |

**Суть:** Команда `pip install -r requirements.txt` одинакова в обеих ветках `if/else`. Разница только в том, что при создании нового venv дополнительно обновляется pip.

**Рекомендация:** Вынести общую часть после `if`:
```bash
if [[ ! -d .venv ]]; then
  python3 -m venv .venv
  .venv/bin/pip install --upgrade pip
fi
.venv/bin/pip install -r requirements.txt
```

## 3. Переусложнённые абстракции

### 3.1 `Field(...)` с description в NotifyRequest — избыточны для внутреннего сервиса

| Файл | Строки |
|---|---|
| `alertsbot/app.py` | 17–20 |

**Суть:** Все четыре поля `NotifyRequest` обёрнуты в `Field(..., description="...")`. Descriptions попадают в автогенерируемые OpenAPI-доки (`/docs`). Для внутреннего alerting-сервиса, к которому обращаются по документации, Swagger-описания вряд ли кто-то открывает.

**Рекомендация:** Можно упростить до голых аннотаций:
```python
class NotifyRequest(BaseModel):
    service: str
    title: str
    message: str
    details: str | None = None
```
Поведение валидации не изменится. Если OpenAPI-доки нужны — оставить как есть.

## 4. Функции, которые можно заинлайнить

**Не обнаружено.** В проекте ровно три модуля, каждая функция вызывается ровно из одного места, но инлайнинг ухудшит читаемость и тестируемость:

- `get_settings()` — нельзя убрать, потому что `@lru_cache` привязан к функции.
- `send_message()` — выделена в отдельный модуль для изоляции httpx-логики от FastAPI. Это правильная граница.

## 5. Устаревшие / избыточные комментарии (WHAT, не WHY)

### 5.1 Docstrings дублируют имена функций

| Файл | Строка | Docstring | Проблема |
|---|---|---|---|
| `alertsbot/app.py` | 31 | `"""Проверка доступности сервиса."""` | Имя `healthz` уже говорит само за себя |
| `alertsbot/app.py` | 41 | `"""Отправляет уведомление в Telegram."""` | Имя `notify` + сигнатура и так передают смысл |
| `alertsbot/telegram.py` | 15 | `"""Отправляет сообщение в Telegram."""` | Дословно повторяет имя `send_message` |
| `alertsbot/config.py` | 31 | `"""Возвращает кэшированные настройки."""` | Имя `get_settings` + `@lru_cache` видны |

**Рекомендация:** Либо удалить, либо заменить на WHY-комментарии там, где есть неочевидное поведение. Например, для `notify` полезнее был бы комментарий про `# noqa: BLE001` — почему ловим голый Exception (ответ: httpx может кинуть любой из нескольких типов исключений, и все они должны стать 502).

### 5.2 Модульные docstrings

| Файл | Строка | Docstring |
|---|---|---|
| `alertsbot/__init__.py` | 1 | `"""Alertsbot package."""` |
| `alertsbot/config.py` | 1 | `"""Конфигурация alertsbot."""` |
| `alertsbot/telegram.py` | 1 | `"""Клиент Telegram для отправки сообщений."""` |

**Суть:** Дублируют имя файла. Для проекта из трёх модулей это не помогает навигации.

**Рекомендация:** Не критично. Можно удалить для чистоты или оставить — вреда нет.

## 6. Мелочи (не баги, но можно подчистить)

### 6.1 Явная type-аннотация для локального dict

| Файл | Строка |
|---|---|
| `alertsbot/telegram.py` | 23 |

**Суть:** `client_kwargs: dict[str, object] = {...}` — аннотация типа для локальной переменной, которая используется через две строки и нигде больше. Тип очевиден из правой части.

**Рекомендация:** Убрать аннотацию: `client_kwargs = {"timeout": timeout, "trust_env": False}`.
